## Overview

This document defines the **mandatory structure** for every phase file generated by the `plan` workflow. A phase file is a compiled prompt — all rules, constraints, context, and paths are pre-resolved and inlined. The executing agent needs **zero external knowledge** to complete the phase.

Phase files are stored in `{cypilot_path}/.plans/{task-slug}/phase-{NN}-{slug}.md`.

---

## Key Invariant

**A phase file MUST be executable by any AI agent without Cypilot context.**

This means:
- Zero unresolved `{variable}` references
- Zero external file references (no "open file X" or "read Y")
- Zero Cypilot-specific terminology without inline definition
- All rules inlined verbatim — not referenced by path
- All input content inlined or quoted — not referenced by path
- Binary acceptance criteria — pass/fail, no subjective judgment

---

## Template Structure

Phase files MUST contain exactly these sections in this order. No sections may be added, removed, or reordered.

### Section 1: TOML Frontmatter

Fenced TOML block at the very start of the file. Machine-readable metadata for the plan manifest.

```toml
[phase]
plan = "{task-slug}"
number = {N}
total = {M}
type = "{generate|analyze|implement}"
title = "{short descriptive title}"
depends_on = []        # list of phase numbers that must complete first
input_files = []       # project files this phase reads (must exist before execution)
output_files = []      # project files this phase creates or modifies
outputs = []           # intermediate results this phase produces for later phases (in out/)
inputs = []            # intermediate results from prior phases this phase needs (from out/)
```

**Field rules**:
- `plan` — slug matching the parent directory name in `.plans/`
- `number` — 1-indexed, sequential, unique within the plan
- `total` — total number of phases in the plan
- `type` — one of: `generate` (create/update artifacts), `analyze` (validate/review), `implement` (write code)
- `depends_on` — empty list `[]` if no dependencies; otherwise phase numbers (integers)
- `input_files` — project files (absolute or project-relative paths); must exist before execution
- `output_files` — project files (absolute or project-relative paths); created or modified by this phase
- `outputs` — intermediate result files in `out/` that later phases depend on (data contract)
- `inputs` — intermediate result files from prior phases that this phase reads from `out/`

### Section 2: Preamble

Fixed text that establishes agent independence. MUST be included verbatim:

```markdown
## Preamble

This is a self-contained phase file. All rules, constraints, input content,
and tool commands are included below. You do not need any prior context or
external files. Follow the instructions exactly, run any EXECUTE commands
as written, and report results against the acceptance criteria at the end.
```

### Section 3: What

One paragraph (2-5 sentences) describing what this phase accomplishes. MUST include:
- The concrete deliverable (what file is created or what check is performed)
- The scope boundary (what is IN and OUT of this phase)

```markdown
## What

{Describe the concrete deliverable and scope. Example:}
Create the "Overview" and "Actors" sections of the PRD artifact at
`architecture/PRD.md`. This phase covers sections 1-3 of the template.
Sections 4+ are handled by subsequent phases. Do NOT create or modify
any other files.
```

### Section 4: Prior Context

Summary of relevant decisions, constraints, or outputs from previous phases. For phase 1, this section contains project-level context. For later phases, it includes a brief summary of what earlier phases produced.

```markdown
## Prior Context

{For phase 1: project context, tech stack, key decisions.}
{For phase N>1: "Phase {N-1} created X. Key decisions: ..."}
{If no prior context needed: "No prior context required."}
```

**Rules**:
- Maximum 20 lines
- Facts only — no opinions or suggestions
- Reference output_files from prior phases if relevant

### Section 4b: User Decisions (optional)

Present ONLY if this phase has user interaction points detected during plan generation. Lists pre-resolved decisions (already answered) and phase-bound questions (to ask during execution).

````markdown
## User Decisions

### Already Decided (pre-resolved during planning)

- **System slug**: `ex-ovwa` — used in all ID patterns
- **Tech stack**: Python 3.12+, stdlib only
- **Architecture style**: modular monolith

### Decisions Needed During This Phase

- [ ] Review the generated sections before writing to disk (present output first)
- [ ] Confirm actor list is complete before proceeding to requirements
````

**Rules**:
- Pre-resolved decisions MUST be inlined as facts, not as questions
- Phase-bound questions MUST be phrased as actionable items with `- [ ]` checkboxes
- Review gates MUST appear as explicit steps: "Present output to user for review"
- If no interaction points exist for this phase, omit this section entirely
- Maximum 30 lines

### Section 5: Rules

All applicable rules from the kit's `rules.md` (and `checklist.md` for analyze tasks) inlined **verbatim and completely**. These are the constraints the agent MUST follow while executing the task.

> **Kit rules are the highest-priority content in a phase file.** Every MUST/MUST NOT rule from `rules.md` that applies to this phase's artifact kind and scope MUST be present. Rules are NEVER summarized, trimmed, or selectively excerpted. Completeness overrides the line budget.

```markdown
## Rules

{ALL applicable rules from rules.md, inlined verbatim. Example:}

### Structure Rules
- Every section MUST have exactly one H2 heading
- IDs MUST follow the pattern `cpt-{system}-{kind}-{slug}`
- ...

### Content Rules
- Descriptions MUST use imperative mood
- ...

### Forbidden
- MUST NOT include implementation details
- MUST NOT reference code files
- ...
```

**Rules for the Rules section**:
- Inline ALL rules from `rules.md` that apply to this phase's scope — **no omissions**
- Group by category (Structure, Content, Forbidden, Style)
- Use MUST/MUST NOT language — no "should" or "consider"
- Target ≤ 200 lines, but **completeness overrides the budget**
- If rules exceed 300 lines: do NOT trim rules — split the phase into sub-phases with narrower scope, each carrying the full rules for its narrower scope
- After all phases are compiled, the union of all Rules sections MUST cover 100% of `rules.md`

### Section 6: Input

All input content the agent needs, inlined verbatim. This includes template sections, existing file content to modify, reference data, or examples.

````markdown
## Input

{Inlined content. Example:}

### Template (sections 1-3)

```markdown
# {Project Name} — Product Requirements Document
## 1. Overview
...
## 3. Actors
...
```

### Existing Content (if editing)

```markdown
{Current file content that this phase modifies}
```

### Reference Example

```markdown
{Example of a completed artifact for reference}
```
````

**Rules**:
- ALL input MUST be inlined — no file paths to read
- Template variables MUST be pre-resolved before inlining
- If input exceeds 300 lines, the phase scope is too broad — split it
- Use fenced code blocks with language hints for all inlined content

### Section 7: Task

Step-by-step instructions the agent follows. Each step is a concrete action with a verifiable outcome.

```markdown
## Task

1. Create file `{output_path}` with the TOML frontmatter from the template
2. Write section 1 (Overview) following the Structure Rules above
3. Write section 2 (Problem Statement) using the Prior Context
4. Write section 3 (Actors) — define each actor with role and description
5. Verify all IDs match the pattern in Rules
6. Count total lines — must be under {budget} lines for this section
```

**Rules**:
- 3-10 steps per phase (fewer = too vague, more = too complex)
- Each step produces a visible result (file content, validation output)
- Steps reference sections by name ("the Structure Rules above"), not by file path
- No conditional branching — if conditions exist, split into separate phases
- Final step MUST be self-verification against acceptance criteria

### Section 8: Acceptance Criteria

Binary pass/fail checks. The agent evaluates each criterion after completing the task and reports the result.

```markdown
## Acceptance Criteria

- [ ] File `{output_path}` exists and is valid Markdown
- [ ] Section 1 (Overview) contains project name and one-paragraph description
- [ ] Section 2 (Problem Statement) lists at least 2 problems
- [ ] Section 3 (Actors) defines at least 1 actor with ID matching `cpt-{system}-actor-{slug}`
- [ ] All IDs follow the pattern `cpt-{system}-{kind}-{slug}`
- [ ] Total line count ≤ {budget}
- [ ] No unresolved template variables (`{...}` patterns)
```

**Rules**:
- 3-10 criteria per phase
- Each criterion is objectively verifiable (exists, matches pattern, count ≤ N)
- No subjective criteria ("well-written", "clear", "appropriate")
- MUST include a line count check if budget enforcement applies
- MUST include a "no unresolved variables" check

### Section 9: Output Format

Fixed format for the agent's completion report. MUST be included verbatim:

````markdown
## Output Format

When complete, report results in this exact format:

```
PHASE {N}/{M} COMPLETE
Status: PASS | FAIL
Files created: {list}
Files modified: {list}
Acceptance criteria:
  [x] Criterion 1 — PASS
  [ ] Criterion 2 — FAIL: {reason}
  ...
Line count: {actual}/{budget}
Notes: {any issues or decisions made}
```

Then generate a **copy-pasteable prompt** for the next phase inside a single code fence:

```
Next phase prompt (copy-paste into new chat if needed):
```

```
I have a Cypilot execution plan at:
  {cypilot_path}/.plans/{task-slug}/plan.toml

Phase {N} is complete ({status}).
Please read the plan manifest, then execute Phase {N+1}: "{next_title}".
The phase file is: {cypilot_path}/.plans/{task-slug}/{next_phase_file}
It is self-contained — follow its instructions exactly.
After completion, report results and generate the prompt for the next phase.
```

If this is the **last phase**, instead of a next-phase prompt output:

```
ALL PHASES COMPLETE ({M}/{M})
Plan: {cypilot_path}/.plans/{task-slug}/plan.toml
Lifecycle: {lifecycle_strategy}
```

Then ask: `Continue in this chat? [y/n]`
````

## Compilation Rules

When the plan workflow compiles a phase file, it MUST:

1. **Resolve all variables** — replace every `{variable}` with its concrete value from `cpt info` output or plan context
2. **Inline all rules** — copy relevant rules from `rules.md` and `checklist.md` verbatim into Section 5
3. **Inline all input** — copy template sections, existing content, and examples verbatim into Section 6
4. **Pre-resolve paths** — replace all relative paths with project-relative paths
5. **Strip irrelevant content** — only include rules and input relevant to this phase's scope
6. **Validate no external references** — scan for patterns like "open file", "read", "see {path}" and replace with inlined content
7. **Validate no unresolved variables** — scan for `{...}` patterns not inside code fences; zero allowed

---

## Budget Enforcement

| Metric | Target | Maximum | Action if exceeded |
|--------|--------|---------|-------------------|
| Total phase file lines | ≤ 500 | ≤ 1000 | Split into sub-phases |
| Rules section (Section 5) | ≤ 200 | ≤ 300 | Reduce rule scope |
| Input section (Section 6) | ≤ 300 | ≤ 500 | Split input across phases |
| Task steps (Section 7) | 3-7 | 10 | Split task |
| Acceptance criteria (Section 8) | 3-7 | 10 | Merge related criteria |

If a compiled phase exceeds 1000 lines, the plan workflow MUST split it into sub-phases automatically.

---

## Anti-Patterns

| Anti-Pattern | Example | Fix |
|-------------|---------|-----|
| External reference | "Read rules from `rules.md`" | Inline the rules verbatim |
| Unresolved variable | "Create file at `{output_path}`" | Resolve to concrete path |
| Subjective criterion | "Code is well-structured" | "All functions ≤ 30 lines" |
| Conditional branching | "If X, do A; else do B" | Split into two phases |
| Scope creep | Phase covers 5+ template sections | Split into 2-3 phases |
| Missing preamble | No self-containment statement | Include verbatim preamble |
| Cypilot jargon | "Run `cpt validate`" without context | Provide full command with args |

## Example Phase File

````markdown
```toml
[phase]
plan = "generate-prd-overwork-alert"
number = 1
total = 4
type = "generate"
title = "PRD Overview, Problem Statement, and Actors"
depends_on = []
input_files = []
output_files = ["examples/overwork_alert/architecture/PRD.md"]
outputs = ["out/phase-01-actors.md"]
inputs = []
```

## Preamble

This is a self-contained phase file. All rules, constraints, input content,
and tool commands are included below. You do not need any prior context or
external files. Follow the instructions exactly, run any EXECUTE commands
as written, and report results against the acceptance criteria at the end.

## What

Create the initial PRD file for the Overwork Alert project at
`examples/overwork_alert/architecture/PRD.md` with sections 1-3:
Overview, Problem Statement, and Actors. Sections 4+ (Requirements,
Use Cases, Acceptance Criteria) are handled by phases 2-4.
Do NOT create or modify any other files.

## Prior Context

Project: Overwork Alert — a macOS daemon that monitors user activity
and sends notifications when work sessions exceed configured thresholds.
Tech stack: Python 3.12+, stdlib only, launchd for scheduling.
Target users: individual developers who tend to overwork.

## Rules

### Structure Rules
- File MUST start with YAML frontmatter containing cypilot markers
- H1 title MUST match format: `{Project Name} — Product Requirements Document`
- Each section MUST have exactly one H2 heading
- IDs MUST follow pattern: `cpt-{system}-{kind}-{slug}`
- System slug for this project: `ex-ovwa`

### Content Rules
- Overview MUST be 2-5 sentences describing the product
- Problem Statement MUST list 2+ concrete problems with evidence
- Each actor MUST have: ID (as task checkbox), name, and role description
- Actor IDs MUST use kind `actor`: `cpt-ex-ovwa-actor-{slug}`

### Forbidden
- MUST NOT include implementation details or technical architecture
- MUST NOT reference code files or specific modules
- MUST NOT use passive voice in problem descriptions

## Input

### Template (sections 1-3)

```markdown
# {Project Name} — Product Requirements Document

- **System**: {system_name}
- **ID Prefix**: `cpt-{system_slug}`
- **Version**: v1

## 1. Overview

{2-5 sentences describing the product, its purpose, and target audience.}

## 2. Problem Statement

{List concrete problems this product solves. Each problem should be
evidence-based and measurable where possible.}

## 3. Actors

| Actor | Description |
|-------|-------------|
| `cpt-{system}-actor-{slug}` | {role description} |
```

## Task

1. Create file `examples/overwork_alert/architecture/PRD.md`
2. Add YAML frontmatter with `cypilot: true`, `type: artifact`, `kind: PRD`
3. Write H1 title: `Overwork Alert — Product Requirements Document`
4. Write section 1 (Overview): describe the product in 3-4 sentences
5. Write section 2 (Problem Statement): list 3 concrete problems developers face with overwork
6. Write section 3 (Actors): define User, Daemon, and LaunchAgent Manager actors with IDs
7. Save actor list (IDs + names) to `cypilot/.plans/generate-prd-overwork-alert/out/phase-01-actors.md`
8. Self-verify against acceptance criteria below

## Acceptance Criteria

- [ ] File `examples/overwork_alert/architecture/PRD.md` exists
- [ ] YAML frontmatter contains `cypilot: true` and `kind: PRD`
- [ ] H1 matches `Overwork Alert — Product Requirements Document`
- [ ] Section 1 contains 2-5 sentences
- [ ] Section 2 lists at least 2 problems
- [ ] Section 3 defines at least 2 actors with IDs matching `cpt-ex-ovwa-actor-*`
- [ ] All IDs follow pattern `cpt-ex-ovwa-{kind}-{slug}`
- [ ] No unresolved `{...}` template variables outside code fences
- [ ] Total line count ≤ 120
- [ ] File `out/phase-01-actors.md` exists and lists all actor IDs

## Output Format

When complete, report results in this exact format:

```
PHASE 1/4 COMPLETE
Status: PASS | FAIL
Files created: examples/overwork_alert/architecture/PRD.md, out/phase-01-actors.md
Files modified: none
Acceptance criteria:
  [x] Criterion 1 — PASS
  ...
Line count: {actual}/120
Notes: {any issues or decisions made}
```

Next phase prompt (copy-paste into new chat if needed):

```

Please read the plan manifest, then execute Phase 2: "PRD Goals and Operational Concept".

It is self-contained — follow its instructions exactly.
After completion, report results and generate the prompt for the next phase.
```
````
